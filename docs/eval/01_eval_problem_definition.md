# 왜 평가를 다시 설계해야 했는가

## 목적

이 문서는 배달 앱 AI 상담사 파인튜닝 프로젝트에서 왜 별도의 평가 체계를 다시 설계했는지 설명하는 문제 정의 문서다.

이 문서의 역할은 세 가지다.

- 합성 데이터 생성만으로는 충분하지 않았던 이유를 설명한다.
- 데이터 검증과 모델 평가를 왜 분리해서 다뤘는지 정리한다.
- 이후 문서인 `02_eval_metric_report.md`, `03_eval_plan.md`, `04_metric_issue_resolution_workflow.md`가 어떤 문제를 해결하기 위해 생겼는지 연결한다.

## 출발점

이 프로젝트는 OpenAI Batch API로 멀티턴 function calling 데이터를 생성하고, 이를 QLoRA SFT 학습에 사용한 뒤 tool calling 성능을 평가하는 파이프라인이다.

초기에는 "좋은 데이터를 만들면 좋은 모델이 나온다"는 가정으로 출발했지만, 실제로는 그 사이에 두 개의 간극이 있었다.

1. 생성된 데이터가 구조적으로 올바른가
2. 구조적으로 올바른 데이터가 실제 모델 성능 향상으로 이어지는가

첫 번째 질문만 통과하고 두 번째 질문을 놓치면, 겉보기에는 깨끗한 데이터로 학습했는데도 실제 추론 단계에서는 함수 선택, 인자 보존, 멀티턴 상태 추적에서 실패할 수 있다.

## 왜 합성 데이터만 믿을 수 없었는가

합성 데이터는 빠르게 대량 생성할 수 있지만, 그대로 학습 데이터로 사용하기에는 네 가지 위험이 있었다.

- 구조 오류: `<tool_call>` 형식, 함수명, 파라미터 타입, required 인자 누락 같은 오류가 학습 데이터에 그대로 들어갈 수 있다.
- 규칙 오염: 함수 시그니처는 맞더라도, `search_restaurants`처럼 호출 규약이 과하게 설계되면 모델이 불필요한 슬롯까지 학습한다.
- 의미 품질 문제: 자연스러운 대화처럼 보여도 실제로는 고객 의도와 어긋난 tool 선택이나 부자연스러운 clarification이 섞일 수 있다.
- 다운스트림 불확실성: 데이터 자체는 그럴듯해 보여도, 실제 파인튜닝 후 성능이 좋아졌는지는 별도 검증 없이는 알 수 없다.

즉 "합성 데이터 생성"과 "합성 데이터의 유효성 증명"은 다른 문제였다.

## 그래서 평가를 세 층으로 나눴다

초기 조사 단계에서 정리한 핵심 판단은 평가를 하나의 점수로 뭉뚱그리면 안 된다는 것이었다. 이 프로젝트에서는 평가를 아래 세 층으로 분리해 다루는 것이 더 적절했다.

### 1. 데이터 구조 검증

가장 먼저 확인해야 할 것은 데이터가 함수 계약과 대화 포맷을 깨지 않는지다.

현재 프로젝트에서는 이 역할을 `datavalidator`가 담당한다.

- `datagen/tool_specs.py`를 단일 원본으로 사용
- tool name, parameter, type, required 여부 검증
- ChatML 샘플의 role 순서와 형식 검증

이 단계는 "학습 가능한 데이터인가"를 판단하는 최소 관문이다.

### 2. 데이터 의미 품질 검토

구조가 맞는다고 해서 곧바로 좋은 데이터는 아니다.

예를 들어 아래와 같은 문제는 구조 검증만으로 잡히지 않는다.

- 사용자가 주문번호를 주지 않았는데도 `get_order_status`를 호출하는 케이스
- soft preference를 숫자 threshold로 과해석해 `min_rating`을 붙이는 케이스
- 정책 안내가 우선이어야 하는데 불필요한 tool call을 생성하는 케이스

이 층에서는 "이 대화가 실제로 올바른 라벨링인가"를 본다. 이후 트러블슈팅 과정에서 발견된 다수의 문제는 바로 이 층의 규칙 결함이었다.

### 3. 다운스트림 모델 평가

최종적으로 중요한 것은 파인튜닝된 모델이 실제 evaluation dataset에서 어떻게 동작하는지다.

이 역할은 `evaluations`가 담당한다.

- tool relevance detection
- function matching
- parameter hallucination / missing / type / value 평가
- turn level / conversation level 집계

이 단계가 필요한 이유는, 데이터 검증을 통과한 데이터셋이라도 실제 모델은 멀티턴에서 다른 실패 양상을 보일 수 있기 때문이다.

## 데이터 검증과 모델 평가는 왜 분리되어야 했는가

이 프로젝트에서 가장 중요한 설계 판단 중 하나는 "데이터 검증"과 "모델 평가"를 같은 문맥으로 설명하되, 실제 운영 단계에서는 분리하는 것이었다.

그 이유는 다음과 같다.

- 데이터 검증은 생성 파이프라인의 품질 관리 문제다.
- 모델 평가는 학습 결과물의 성능 진단 문제다.
- 둘을 섞으면 실패 원인이 데이터셋 라벨 문제인지, 함수 규약 문제인지, 모델 자체의 한계인지 분해하기 어렵다.

실제 트러블슈팅에서 이 분리는 매우 중요했다. 평가 지표가 하락했을 때 곧바로 "모델이 못한다"로 결론내리지 않고, 아래 질문을 순서대로 분리할 수 있었기 때문이다.

1. eval dataset gold가 현재 규칙과 맞는가
2. train data가 잘못된 규칙을 학습시키고 있는가
3. `datagen/tool_specs.py`와 프롬프트 규약이 과설계되어 있는가
4. 위를 정리해도 남는 오차가 모델 자체 한계인가

이 사고 구조는 이후 `04_metric_issue_resolution_workflow.md`의 기반이 되었다.

## 왜 외부 벤치마크를 그대로 쓰지 않았는가

초기 조사에서는 BFCL, HammerBench, Unitxt, TSTR를 폭넓게 검토했다. 그러나 이 프로젝트는 그중 하나를 그대로 채택하지 않고, 각 프레임워크에서 필요한 관점만 가져왔다.

이유는 프로젝트 제약이 분명했기 때문이다.

- 도메인은 범용 agent가 아니라 배달 앱 상담 시나리오다.
- parallel call보다 sequential call이 중요하다.
- 실제 백엔드 상태를 재현하는 실행 환경이 아니라, GT history 기반 비교가 더 현실적이다.
- 목표는 리더보드 점수 자체보다 실패 원인을 분해해 데이터와 규칙을 개선하는 것이다.

그래서 최종 방향은 아래처럼 정리되었다.

- BFCL에서: 구조 정확성과 멀티턴 평가 관점을 참고
- HammerBench에서: 턴 단위 오류 분해와 진단 친화성 참고
- Unitxt에서: parameter 수준 분해 메트릭 관점 참고
- TSTR에서: "합성 데이터가 실제 성능에 기여하는가"라는 검증 철학 참고

이 비교와 선택의 배경은 `02_eval_metric_report.md`에 정리되어 있다.

## 이 문제 정의가 실제로 필요했던 이유

이 문서가 단순 배경 설명이 아니라 실제 작업에 필요했던 이유는, 평가 체계가 이후 트러블슈팅과 데이터 수정의 기준이 되었기 때문이다.

실제 프로젝트에서는 다음과 같은 흐름이 반복되었다.

1. 평가 지표에서 실패 축을 발견한다.
2. 실패를 함수 선택, relevance, parameter value 문제로 분해한다.
3. 실패가 eval_data, train_data, datagen 규약, 모델 중 어디서 왔는지 판단한다.
4. 수정 후 다시 평가해 개선 여부를 확인한다.

즉 평가는 최종 보고용 점수가 아니라, 데이터와 규칙을 수정하기 위한 진단 도구였다.

2. `02_eval_metric_report.md`
   어떤 외부 평가 프레임워크를 검토했고, 무엇을 참고했는지 정리한다.
3. `03_eval_plan.md`
   이 프로젝트가 최종적으로 채택한 평가 설계를 설명한다.
4. `04_metric_issue_resolution_workflow.md`
   평가 결과를 실제 데이터/규칙 수정 루프로 연결하는 절차를 설명한다.

이 문서는 평가 체계의 문제 정의와 설계 배경을 다룬다. 구체적인 metric 정의, 멀티턴 분할 방식, 스코어링 규칙은 `03_eval_plan.md`에서 다룬다.
