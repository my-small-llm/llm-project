# `search_restaurants` 스키마 단순화 결정 기록

작성일: 2026-03-28

## 문서 목적

이 문서는 `search_restaurants` 관련 스키마 변경 결정을 한곳에 정리한 통합 기록이다.

이번 정리는 훈련 데이터와 평가 품질을 높이기 위해, 사용자 의도와 직접 연결되지 않거나 의미가 중복되는 파라미터를 단순화하는 데 목적이 있다.

검토 대상은 아래 다섯 항목이었다.

- `page`
- `page_size`
- `only_open`
- `sort`
- `min_rating`

## 공통 배경

기존 데이터셋과 실패 보고서를 함께 보면 `search_restaurants`는 실제 사용자 의도를 표현하는 핵심 슬롯과, 백엔드 기본 정책이나 데이터 생성 관성 때문에 호출 인자에 노출된 슬롯이 섞여 있었다.

특히 아래 문제가 반복됐다.

- 사용자가 말하지 않은 기본값이 호출 인자에 들어감
- 생략으로 충분한 값이 `false` 또는 `null`로 과하게 명시됨
- 모호한 요청이 임의의 구체값으로 변환됨
- 핵심 검색 의도는 맞았는데 보조 인자 때문에 `argument_value` 실패가 발생함

이 때문에 `search_restaurants`를 “사용자 검색 의도 표현” 중심으로 다시 정리하기로 했다.

## 항목별 결정

### 1. `page`, `page_size`

#### 결정

호출 인자에서 제거한다.

#### 이유

- 현재 데이터셋에서 사용자 발화에 직접 등장하는 경우보다, 응답 메타데이터나 기본 페이지네이션 정책처럼 붙는 경우가 많았다.
- 실패 보고서에서 `page/page_size` 생략, 기본값 회귀 같은 exact match 실패를 자주 만들었다.
- 현재 프로젝트 목적에서는 페이지 탐색보다 검색 의도 표현의 안정성이 더 중요했다.

#### 변경 후 정책

- 검색 결과는 항상 `1페이지`부터 시작한다.
- 한 페이지에 보여줄 개수는 백엔드 고정 정책으로 처리한다.
- 응답의 `pagination` 메타데이터는 유지할 수 있다.

#### 예시

- 변경 전
  - 사용자: `마라탕집 찾아줘`
  - 호출: `search_restaurants(query="마라탕", page=1, page_size=20)`
- 문제점
  - 사용자가 페이지나 개수를 말하지 않았는데도 기본값이 호출 인자에 노출됐다.
  - 모델은 검색 의도보다 기본 pagination 값까지 맞춰야 해서 불필요한 오류가 늘어났다.
- 변경 후
  - 호출: `search_restaurants(query="마라탕")`
  - 설명: 검색은 항상 1페이지부터 시작하고 개수는 백엔드가 고정 정책으로 처리

### 2. `only_open`

#### 결정

유지하되, `true 또는 생략` 규약으로 단순화한다.

#### 이유

- `only_open` 자체는 사용자 의도와 실제로 자주 연결되는 핵심 필터다.
- 하지만 기존 데이터는 `true` 편향이 강했고, `false`와 생략의 경계가 모호했다.
- 실패 보고서에서도 `false/미지정 -> true` 회귀와 이전 턴 조건 누수가 반복됐다.

#### 변경 후 규약

- 사용자가 `영업 중`, `지금 열려 있는`, `문 연 곳만`처럼 명시하면 `only_open=true`
- 그 외에는 `only_open` 생략
- `only_open=false`는 호출 인자에서 적극적으로 사용하지 않음

#### 예시

- 변경 전
  - 사용자: `치킨집 찾아줘`
  - 호출: `search_restaurants(category="치킨", only_open=False)`
- 문제점
  - `only_open=False`는 생략과 의미가 거의 같아서, 모델이 `false`, 생략, `true`를 불필요하게 혼동했다.
- 변경 후
  - 호출: `search_restaurants(category="치킨")`
  - 설명: 영업 여부를 특정하지 않았으므로 `only_open` 생략

- 참고
  - 사용자: `지금 영업 중인 치킨집 찾아줘`
  - 호출: `search_restaurants(category="치킨", only_open=True)`
  - 설명: 이 경우는 규약 변경 이후에도 그대로 유효한 정상 사용 사례

### 3. `sort`

#### 결정

유지하되, 사용자가 정렬 기준을 명시했을 때만 호출 인자에 포함한다.

#### 이유

- 기존 데이터에서는 `sort="rating"` 편향이 매우 강했다.
- 사용자가 정렬 기준을 말하지 않아도 `rating`이 기본값처럼 자주 붙었다.
- 실패 보고서에서는 `relevance`와 `rating` 혼동이 반복됐다.

#### 변경 후 규약

- `평점순`, `별점 높은 순` -> `sort="rating"`
- `배달비 낮은 순`, `배달요금 싼 순`, `무료 배달 우선` -> `sort="delivery_fee"`
- `관련도순` -> `sort="relevance"`
- 정렬 기준 언급이 없으면 `sort` 생략

#### 백엔드 기본 정책

- `sort`가 생략되면 백엔드는 기본적으로 `rating` 기준으로 결과를 반환한다.

#### 예시

- 변경 전
  - 사용자: `치킨집 찾아줘`
  - 호출: `search_restaurants(category="치킨", sort="rating")`
- 문제점
  - 사용자가 정렬 기준을 말하지 않았는데도 `rating`이 기본값처럼 호출 인자에 들어갔다.
  - 이 때문에 `relevance`와 `rating` 혼동, 불필요한 exact match 실패가 발생했다.
- 변경 후
  - 호출: `search_restaurants(category="치킨")`
  - 설명: 정렬 기준을 직접 말하지 않았으므로 `sort`는 생략하고, 백엔드 기본 정렬 적용

- 참고
  - 사용자: `치킨집 평점순으로 찾아줘`
  - 호출: `search_restaurants(category="치킨", sort="rating")`
  - 설명: 정렬 기준이 명시된 경우에는 지금도 `sort`를 명시하는 것이 맞음

### 4. `min_rating`

#### 결정

유지하되, 명시적 숫자 기준 필터로만 사용한다.

#### 이유

- `min_rating` 자체는 필요하지만, 기존 데이터에서는 `평점 높은 곳` 같은 모호한 표현이 `4.5`, `4.6` 같은 임의 기준으로 번역되는 경우가 있었다.
- 일부 샘플에서는 `min_rating: null`이 불필요하게 호출 인자에 들어갔다.
- 이 때문에 `평점 높은 곳`과 `평점 4.5 이상`이라는 서로 다른 의도가 구분되지 않았다.

#### 변경 후 규약

- `4.5 이상`, `최소 4.3`, `4점 넘는 곳`처럼 숫자 기준이 있으면 `min_rating` 사용
- `평점 높은 곳`, `별점 좋은 곳`처럼 모호하면 `min_rating` 생략
- `min_rating: null`은 호출 인자에서 사용하지 않음

#### 품질 선호 처리

- 숫자 기준 없는 품질 선호는 `sort="rating"` 또는 백엔드 기본 정렬로 처리한다.

#### 예시

- 변경 전
  - 사용자: `평점 높은 치킨집 찾아줘`
  - 호출: `search_restaurants(category="치킨", min_rating=4.5, sort="rating")`
- 문제점
  - 사용자가 숫자 기준을 말하지 않았는데도 `4.5` 같은 임의 기준이 들어가, 모호한 선호가 수치 임계값으로 왜곡됐다.
- 변경 후
  - 호출: `search_restaurants(category="치킨", sort="rating")`
  - 설명: 숫자 기준이 없으므로 `min_rating`는 쓰지 않음

- 참고
  - 사용자: `평점 4.5 이상인 치킨집 찾아줘`
  - 호출: `search_restaurants(category="치킨", min_rating=4.5)`
  - 설명: 숫자 기준을 직접 말한 경우에는 지금도 `min_rating`를 사용하는 것이 맞음

## 최종 호출 규약

현재 `search_restaurants`는 아래 원칙으로 호출한다.

- `query`
  - 사용자가 식당명이나 메뉴 키워드를 말했을 때 사용
- `category`
  - 사용자가 카테고리를 지정했을 때 사용
- `min_rating`
  - 숫자 임계값을 직접 말했을 때만 사용
- `only_open`
  - 영업 중 조건을 직접 말했을 때만 `true`
- `sort`
  - 정렬 기준을 직접 말했을 때만 사용

반대로 아래 값은 모델 호출 인자에서 사용하지 않는다.

- `page`
- `page_size`
- `only_open=false`
- `min_rating=null`
- 정렬 기준이 없는 상황의 `sort`

## 기대 효과

- 사용자 발화와 함수 인자의 대응 관계가 더 명확해진다.
- 백엔드 기본 정책과 사용자 의도 슬롯이 분리된다.
- 불필요한 기본값, `false`, `null` 생성이 줄어든다.
- `argument_value` 평가에서 의미 없는 mismatch가 감소한다.
- `search_restaurants`가 핵심 검색 의도 표현에 더 집중하게 된다.

## 요약

이번 변경의 방향은 일관된다.

- 페이지네이션은 백엔드 책임으로 이동
- 영업 여부는 `true 또는 생략`
- 정렬은 명시 요청 시에만 사용
- 최소 평점은 숫자 기준이 있을 때만 사용

즉 `search_restaurants`는 이제 “모델이 불필요한 기본값을 추측하는 함수”가 아니라, “사용자가 실제로 말한 검색 조건만 표현하는 함수”로 단순화된다.
