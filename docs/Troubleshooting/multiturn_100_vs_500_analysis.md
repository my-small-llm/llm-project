## 문제 설정

`LoRA 500`은 학습 데이터 수가 더 많지만, 멀티턴 주요 지표 일부에서 `LoRA 100`보다 소폭 낮다.

비교값:

| 모델 | turn_level_accuracy | conversation_success_rate | conversation_progress_rate | first_failure_turn_avg | error_cascade_rate |
| --- | ---: | ---: | ---: | ---: | ---: |
| LoRA 100 | 55.44% | 15.00% | 56.81% | 0.65 | 51.91% |
| LoRA 500 | 53.74% | 15.00% | 55.33% | 0.50 | 53.68% |

실제 집계값으로 풀어 쓰면:

- `turn_level_accuracy`: `163/294 -> 158/294`
- `conversation_success_rate`: `6/40 -> 6/40`
- `conversation_progress_rate`: 대화별 성공 턴 비율 평균이 `56.81% -> 55.33%`
- `first_failure_turn_avg`: 실패한 34개 대화의 첫 실패 턴 평균이 `0.65 -> 0.50`
- `error_cascade_rate`: 연속 실패 비율이 약 `68/131 -> 73/136`

즉 `LoRA 500`은 완주 대화 수는 늘지 않았고, 전체 턴 성공 수와 대화별 평균 성공 비율이 모두 줄었으며, 실패한 대화들에서는 첫 실패가 더 이른 턴에 나타났고, 한 번 실패한 뒤 다음 턴도 실패할 가능성도 조금 더 높아졌다.

## 먼저 해석해야 할 점

이번 멀티턴 평가는 GT history 기반이다. 따라서 여기서의 하락은 보통 다음 뜻에 가깝다.

- 이전 턴의 실제 오답이 문맥을 망쳐서 무너졌다기보다
- 모델이 특정 턴 패턴, 분기, 슬롯 회수, tool/no-tool 경계 판단을 반복적으로 더 어려워했다

즉 `500개 데이터셋에 잘못된 정보가 섞여서 모델이 오염되었다`는 가능성은 하나의 후보일 뿐이고, 현재 근거상 더 강한 설명은 `데이터 분포가 멀티턴 안정성을 키우는 방향으로 늘지 않았다`는 쪽이다.

## 핵심 원인 가설

### 1. `500셋`은 `100셋`보다 더 좋은 멀티턴 분포가 아니라, 더 짧고 반복적인 흐름이 많다

기존 분석 결과:

- 허브 100셋 평균 tool call: `5.82회`
- 로컬 500셋 평균 tool call: `3.84회`
- 허브 100셋 평균 user 발화: `8.16회`
- 로컬 500셋 평균 user 발화: `4.95회`

또 로컬 500셋은 500개 대화에 대해 tool sequence가 179종뿐이었고, 321개가 이미 본 sequence의 반복이었다.

즉 `500개`는 `100개`보다 더 많은 문장을 포함하지만, 모델이 새롭게 배워야 할 멀티턴 정책 분기 수는 생각보다 많이 늘지 않았다. 오히려 짧은 검색/조회 흐름과 중간 종료 흐름이 더 많이 반복되었다.

이 구조에서는:

- value filling 일부는 보정될 수 있어도
- 대화 초반 분기
- tool/no-tool 게이팅
- 필수 인자 재회수
- 실패 후 다음 턴 복귀

같은 멀티턴 핵심 능력은 충분히 강화되지 않는다.

### 2. `LoRA 500`의 tool-call 경로 진입 안정성이 이미 `LoRA 100`보다 낮다

Tool Call Level 비교에서도 `LoRA 500`은 `LoRA 100`보다 다음이 낮다.

- `relevance_detection_acc`: `86.51% -> 84.47%`
- `required_params_acc`: `99.38% -> 98.67%`
- `argument_type_acc`: `96.89% -> 94.59%`

특히 실제 통과 수는 다음처럼 더 분명하다.

- relevance: `468/541 -> 457/541`
- type 직전 단계까지 살아남은 경로: `161 -> 148`
- 최종 value 정답 수: `96 -> 87`

즉 `LoRA 500`은 value 비율만 보면 근소하게 나쁘지 않지만, 그 전에 올바른 tool-call 경로에 진입하고 유지하는 수 자체가 더 적다.

멀티턴 지표는 결국 턴 pass/fail 집계이므로, 이런 초반 게이팅 약화는 그대로:

- 턴 실패 증가
- 실패 대화에서 더 이른 첫 실패
- 실패 후 연속 실패 증가

로 이어지기 쉽다.

### 3. `500셋`은 end-to-end 주문 완료보다 중간 단계 흐름 비중이 더 높다

허브 100셋은 로컬 500셋보다 다음 비중이 높았다.

- `place_order`
- `prepare_checkout`
- `list_addresses`
- `add_to_cart`

반대로 로컬 500셋은:

- `search_restaurants`
- `get_restaurant_detail`
- 장바구니 진입
- 중간 종료

흐름 비중이 더 높았다.

이 차이는 멀티턴 평가에서 중요하다. 완주형 대화가 많아야 모델이:

- 초반 검색
- 중반 장바구니/주소 회수
- 후반 결제 준비/주문 완료

를 한 대화 안에서 끝까지 연결하는 법을 배우기 쉽다.

반면 짧은 흐름 위주로 늘어난 데이터는 `한 단계씩은 맞히되, 대화 전체를 안정적으로 이어가는 능력`을 충분히 키우지 못한다. 그래서 `conversation_success_rate`는 늘지 않고, 대화별 성공 턴 비율 평균도 오히려 약해질 수 있다.

### 4. no-call과 순수 unsupported 학습 신호가 부족해 초반 분기 실패가 잦아질 수 있다

두 데이터셋 모두 순수 no-call 대화는 사실상 없다. 로컬 500셋은 unsupported 응답이 자주 등장하지만, 대부분은 tool 흐름 중간에 삽입된 형태다.

따라서 모델은 unsupported를:

- `여기서는 끝까지 tool을 쓰지 않는다`

보다

- `잠깐 일반 응답을 하고 다시 tool flow로 돌아간다`

로 배우기 쉽다.

이 편향은 멀티턴 평가에서 특히 초반 턴에 불리하다. 첫 분기에서:

- 이번 턴이 tool 턴인지
- 일반 응답 턴인지
- 이전에 얻은 슬롯을 다시 써야 하는지

를 잘못 판단하면 곧바로 첫 실패가 발생한다.

### 5. `search_restaurants` 같은 고빈도 함수의 인자 분포가 너무 단조롭다

로컬 500셋에서 `search_restaurants`는 매우 자주 등장하지만:

- `only_open`은 사실상 `true`만 등장
- `sort`는 거의 `rating`
- `page`는 전부 `1`
- `page_size`도 대부분 `20`

처럼 정책 다양성이 낮다.

이 경우 모델은 호출 형식과 전형적 값 채우기는 익혀도,

- 애매한 검색 의도
- 여러 후보 중 다시 좁히기
- 이전 턴 제약을 반영한 재검색

같은 멀티턴 분기를 충분히 배우지 못한다.

즉 `argument_value_acc`는 조금 방어할 수 있어도, 멀티턴 지표를 끌어올릴 정도의 정책 일반화는 약해질 수 있다.

## 지표별 해석

### 1. 왜 `turn_level_accuracy`가 `100 > 500`인가

가장 직접적인 이유는 `LoRA 500`이 전체 294턴 중 성공 턴이 5개 적기 때문이다.

이 하락은 다음 조합으로 설명하는 것이 가장 자연스럽다.

- relevance 하락으로 tool/no-tool 경계 판단이 더 흔들렸다
- required/type 하락으로 tool을 골라도 턴 pass까지 이어지는 비율이 낮아졌다
- 추가된 400개가 새로운 분기보다 반복 흐름 중심이라 멀티턴 정책 보강이 약했다

즉 `잘못된 정보가 들어가서 성능이 떨어졌다`는 해석보다, `많아진 데이터가 턴 pass를 늘리는 방향의 신호를 충분히 추가하지 못했다`는 해석이 현재 근거와 더 잘 맞는다.

### 2. 왜 `conversation_progress_rate`가 `100 > 500`인가

`conversation_progress_rate`는 각 대화에서 `맞은 턴 수 / 전체 턴 수`를 계산한 뒤, 그 비율을 대화별로 평균낸 값이다. 따라서 `56.81% -> 55.33%` 하락은 `LoRA 500`이 `LoRA 100`보다 대화 단위 평균 성공 비율이 낮았다는 뜻이다.

이 지표는 `turn_level_accuracy` 하락과 같은 방향의 신호이지만, 전체 턴을 한꺼번에 합쳐 계산한 값이 아니라 각 대화의 성공 비율을 먼저 구한 뒤 평균낸 값이라는 차이가 있다.

### 3. 왜 `first_failure_turn_avg`가 `100 > 500`인가

코드상 `first_failure_turn_avg`는 각 대화에서 처음 실패한 턴의 index를 모아 평균낸 값이다. 성공한 대화는 이 평균 계산에서 제외되며, index는 `0`부터 시작한다. 따라서 값이 낮을수록 `실패한 대화들에서 첫 실패가 더 앞 턴에서 발생했다`는 뜻이다.

따라서 `0.65 -> 0.50` 하락은 `LoRA 500`이 `LoRA 100`보다 실패 대화에서 첫 실패를 더 이른 턴에 맞는 경향이 있었다는 의미다.

이 변화는 다음 가설과 잘 맞는다.

- relevance 약화로 첫 tool/no-tool 분기에서 실수
- 짧고 반복적인 흐름 중심 학습으로 초반 분기 다양성 부족
- unsupported/no-call을 독립 패턴으로 배우지 못해 초반 일반 응답 턴 처리 취약

다만 이 지표는 `실패가 났다면 첫 실패가 몇 번째였는가`를 보는 값이지, 대화 전체 성공 비율을 직접 말해주지는 않는다. 그 부분은 `conversation_progress_rate`와 `turn_level_accuracy`가 따로 설명한다.

### 4. 왜 `error_cascade_rate`가 `100 < 500`로 더 나빠졌는가

코드상 `error_cascade_rate`는 `이전 턴이 실패였던 경우`를 모두 세고, 그중에서 `바로 다음 턴도 실패`한 비율이다. 즉 정확히는 `한 턴이 실패했을 때 다음 턴도 연속해서 실패할 비율`이다.

따라서 `51.91% -> 53.68%` 상승은 `LoRA 500`이 `LoRA 100`보다 한 번 실패한 뒤 다음 턴에서도 다시 실패할 가능성이 조금 더 높았다는 뜻이다.

GT history 평가라는 점을 고려하면, 이 수치는 `이전 오답이 문맥을 오염시켰다`는 뜻이라기보다 `실패가 나온 뒤에도 비슷한 유형의 판단을 연속으로 어려워했다`는 뜻에 가깝다.

이 변화는 다음 가설과 잘 맞는다.

- 실패 이후 복구를 가르치는 데이터가 충분하지 않았다
- 분기 후 복귀, 재검색, 슬롯 재회수 같은 패턴이 부족했다
- 짧고 반복적인 흐름이 많아 연속 실패를 끊는 전략 학습이 약했다

즉 이 수치는 `500셋에 노이즈가 많다`는 직접 증거라기보다, `실패 이후 다음 턴에서 회복하는 멀티턴 패턴`이 충분히 강화되지 않았다는 신호로 해석하는 편이 더 정확하다.

## 가장 가능성이 높은 결론

현재 근거만 보면, `500개 데이터셋에 잘못된 정보가 포함되어 오히려 성능이 떨어졌다`는 가능성을 완전히 배제할 수는 없다. 하지만 더 설득력 있는 설명은 다음 4가지다.

1. `500셋`은 `100셋`의 좋은 분포를 확장한 것이 아니라, 더 짧고 반복적인 흐름을 많이 추가했다.
2. 새로운 데이터가 멀티턴 분기, 복구, no-call 판단, 필수 슬롯 회수 능력을 충분히 늘리지 못했다.
3. Tool Call Level에서 이미 relevance, required, type이 하락했기 때문에 멀티턴에서도 초반 실패와 연속 실패가 늘었다.
4. 현재 `100 vs 500` 비교는 고정 split 실험이 아니어서 데이터 양 효과만의 차이라고 단정하기 어렵다.

따라서 현 시점의 1차 결론은:

- `500이 100보다 나쁜 이유`를 `오답 데이터 오염` 하나로 보기보다
- `분포 차이`, `반복 sequence 증가`, `end-to-end 시나리오 부족`, `복구형 멀티턴 패턴 부족`

의 결합으로 해석하는 것이 더 타당하다.

## 다음 권장 작업

1. 고정된 동일 eval 세트로 `100 vs 500`을 다시 학습해 데이터 양 효과만 분리해서 확인
2. `end-to-end 주문 완료`, `분기 후 복귀`, `필수 슬롯 재회수`, `순수 no-call`, `순수 unsupported` 샘플을 우선 증강
3. `search_restaurants`의 `sort`, `only_open`, `page`, `page_size`, `min_rating` 분포 다양화
4. 실패 대화에서 `한 번 틀린 뒤 다음 턴에서 복구하는` 패턴을 의도적으로 추가

## 한 줄 결론

`LoRA 500`의 멀티턴 하락은 `데이터가 많아졌는데도 더 좋은 대화 정책을 배우지 못한 것`에 가깝고, 현재 증거상 그 원인은 `잘못된 정보 혼입`보다 `좋지 않은 분포 확장`일 가능성이 더 높다.
