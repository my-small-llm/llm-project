# eval_data 대화 품질 점검 문서

## 개요

- 대상: `/home/cwj/llm-project/eval_data/dataset.jsonl`의 대화(`messages`) 부분
- 범위: 대화 흐름, 함수 호출 필요성, 함수 호출 누락/과호출, 공통적인 이상 패턴
- 제외: 툴 스키마 자체의 구현 정확성, 모델 성능 지표, 예시 프롬프트 품질 상세 분석

## 한줄 결론

형식은 전반적으로 잘 맞지만, 학습 데이터로 쓰기에는 몇 개의 충돌 신호가 있습니다. 특히 `복수 의도인데 함수 호출이 하나만 들어간 사례`, `같은 데이터셋 내부 규칙과 충돌하는 응답`, `고객센터형 대화에서의 일관성 부족`은 수정 권장입니다.

## 전체 확인 결과

- 총 40개 대화를 확인함
- 기본 구조는 정상적이었음
- 모든 샘플에서 역할 순서가 크게 깨지지 않았음
- `tool_call`이 있는 경우 다음 턴에 `tool_response`가 오는 형식도 유지됨
- 즉, 포맷 오류보다는 `대화 품질`과 `의사결정 일관성` 문제가 핵심임

## 우선 수정 권장 샘플

### 1. sample_0034: 고객센터형 문의를 과도하게 거절함

파일:
- `/home/cwj/llm-project/eval_data/samples/sample_0034.txt`

문제 구간:
- 35-41행: 라이더 사고 배상 문의, 결제 오류 신고를 모두 `답변할 수 없는 내용`으로 처리

왜 문제인가:
- 같은 데이터셋의 다른 CS 샘플은 사고 접수나 오류 제보에 대해 필요한 정보를 수집하고 담당자 전달 안내를 함
- 그런데 이 샘플은 같은 범주의 문의를 단순 거절함
- 즉, 데이터셋 내부에서 `지원 가능 범위`에 대한 학습 신호가 서로 충돌함

영향:
- 모델이 사고/버그/환불류 문의를 일관되게 처리하지 못할 수 있음
- 어떤 샘플에서는 접수하고, 어떤 샘플에서는 바로 거절하는 식의 흔들림이 생길 가능성이 큼

권장 수정:
- 사고/버그 문의는 무조건 거절하지 말고
- `직접 처리 불가 -> 정보 수집 -> 담당자 전달` 흐름으로 통일하는 편이 좋음

### 2. sample_0038: 복수 의도인데 검색 호출이 하나만 들어감

파일:
- `/home/cwj/llm-project/eval_data/samples/sample_0038.txt`

문제 구간:
- 31행: 사용자가 `치킨`과 `짜장면` 둘 중에서 싸고 빠른 쪽을 골라 달라고 요청
- 33-35행: assistant는 `치킨`에 대해서만 `search_restaurants` 호출
- 40-41행: 짜장면은 아직 안 찾아보고 되묻는 방식으로 응답

왜 문제인가:
- 이 요청은 사실상 두 후보군을 비교하는 성격에 가까움
- 따라서 대화 품질 관점에서는 두 카테고리를 모두 조회한 뒤 비교하는 흐름이 더 자연스러움
- 사용자가 명시적으로 2개의 선택지를 줬는데 실제 함수 호출은 1개만 발생함

영향:
- 모델이 `복수 의도 -> 한 가지만 호출`하는 식으로 학습될 수 있음
- 질문에서 예시로 든 “2개 불러야 하는데 1개만 부른 경우”에 가장 가까운 사례 중 하나임

권장 수정:
- `치킨`, `짜장면` 각각 검색 후 비교하거나
- 최소한 assistant가 `둘 다 비교하려면 두 범주를 모두 조회하겠다`는 식으로 진행하는 샘플로 바꾸는 것이 좋음

### 3. sample_0036: 요청사항 처리 규칙이 데이터셋 내부와 충돌함

파일:
- `/home/cwj/llm-project/eval_data/samples/sample_0036.txt`

문제 구간:
- 87-89행: assistant가 `요청사항은 한 가지만 설정할 수 있어요`라고 말함

왜 문제인가:
- 데이터셋의 다른 주문 샘플들은 하나의 `special_request` 문자열 안에 여러 요청을 합쳐서 전달함
- 예를 들어 `sample_0038`에서는 `면은 조금 덜 삶아 주세요 / 육수는 차갑게 / 고춧가루는 절반만`처럼 복수 요청을 한 번에 저장함
- 즉, 이 샘플 하나가 데이터셋의 암묵적 규칙을 깨고 있음

영향:
- 모델이 어떤 경우에는 복수 요청을 한 번에 넣고
- 어떤 경우에는 `한 가지만 가능`하다고 제한하는 상충된 습관을 학습할 수 있음

권장 수정:
- `한 가지만 설정 가능`이라는 문장을 제거하거나
- 정말 한 가지만 허용하고 싶다면 데이터셋 전체를 그 규칙에 맞게 다시 맞추는 편이 좋음

## 공통적으로 보인 패턴

### 1. 템플릿 반복이 강함

관찰:
- 동일한 마무리 문장 반복이 많음
- 예: `네, 맛있게 드세요! 배달 앱 AI 상담사였습니다. 감사합니다.` 같은 종료 문장이 매우 자주 반복됨

영향:
- 대화 다양성이 낮아짐
- 모델이 특정 문장 패턴에 과도하게 고정될 수 있음

권장:
- 종료 멘트, 완곡 표현, 확인 질문 표현을 더 다양화하는 것이 좋음

### 2. 시나리오 묶음이 너무 뚜렷함

관찰:
- 일부 샘플은 검색만 반복함
- 일부 샘플은 주문상태 조회만 수행함
- 일부 샘플은 고객센터형 비툴 대화만 수행함

영향:
- 데이터셋이 실제 사용자 대화처럼 섞여 있지 않고 묶음형으로 보일 수 있음
- 특정 시나리오에서는 자연스럽지만 전체 데이터 분포로는 다소 인위적임

권장:
- 검색 -> 상세조회 -> 장바구니 -> 결제 확인처럼 단계가 섞인 중간 난도의 대화를 더 넣으면 좋음
- CS 문의와 주문/검색 문의가 섞인 샘플도 일관되게 설계하면 좋음

### 3. 고객센터형 정책이 일관되지 않음

관찰:
- 어떤 샘플은 사고, 환불, 오류 신고를 잘 접수함
- 어떤 샘플은 같은 류 문의를 바로 거절함

영향:
- 모델이 지원 가능 범위를 안정적으로 학습하기 어려움

권장:
- 고객센터형 문의는 다음 중 하나로 통일하는 편이 좋음
- `직접 처리 가능`
- `직접 처리 불가하지만 정보 수집 후 전달 가능`
- `정말 범위 밖이라 안내 불가`

현재는 위 세 가지가 샘플마다 섞여 있음

### 4. 오타/중의적 발화 처리 품질 편차가 있음

관찰:
- 후반부 noisy sample들은 재미있고 실제 사용자스럽지만
- 어떤 샘플은 복수 의도를 잘 풀고
- 어떤 샘플은 한 갈래만 처리하거나 다시 되묻는 비율이 높음

영향:
- noisy input 대응 품질이 일정하지 않게 학습될 수 있음

권장:
- 오타/중의적 발화 샘플은
- `의도 복원 성공`
- `불충분 정보라 확인 질문`
- `복수 의도라 순차 처리`
  중 어떤 전략을 쓸지 더 명확히 맞추는 것이 좋음

## 반대로 괜찮았던 점

- 형식적 파손 샘플은 보이지 않았음
- 대다수 샘플에서 tool 사용 순서는 자연스러웠음
- 주문 플로우 샘플은 전반적으로 읽기 쉬웠음
- noisy user 샘플을 넣어둔 점 자체는 데이터 다양성 측면에서 장점이 있음
