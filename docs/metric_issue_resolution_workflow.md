# Metric Issue Resolution Workflow

## 목적

이 문서는 특정 평가 지표에서 문제를 발견했을 때, 이를 분석하고 실제 수정까지 연결하는 작업 원칙을 정리한 문서다.

이 문서는 사람만 보기 위한 설명 문서가 아니라, AI 에이전트가 읽고 현재 작업 단계를 파악한 뒤 순서대로 실행할 수 있도록 만든 운영 문서이기도 하다.

이 워크플로우의 목표는 아래와 같다.

- 지표 실패를 단순 모델 문제로 뭉뚱그리지 않고 원인을 분해한다.
- 문제 정의와 실제 반영을 분리해서 기록한다.
- 데이터 수정 이력을 추적 가능하게 남긴다.
- 사람과 AI가 같은 기준으로 현재 작업 위치를 공유할 수 있게 한다.

## 기본 원칙

특정 지표에 문제가 보이면 먼저 원인을 아래 범주로 나눠서 본다.

- 평가셋 GT 문제
- 훈련 데이터셋 문제
- datagen 단계의 함수 정의 / 슬롯 규칙 / 프롬프트 문제
- 모델 자체의 문제

하나의 실패가 여러 원인에 걸칠 수 있으므로, 필요하면 복합 원인으로 기록한다.

AI 에이전트는 처음부터 코드나 데이터를 수정하지 말고, 아래 원칙을 따라야 한다.

- 먼저 관련 평가 문서와 실패 사례 문서를 읽는다.
- 문제를 `eval`, `train data`, `datagen`, `model`로 분해한다.
- 순수 모델 실수처럼 보이는 케이스와 실제 수정 가능한 구조 문제를 구분한다.
- 문제 정의 문서를 먼저 만들거나 수정한다.
- 사람 검토를 거쳐 기준이 확정된 뒤에만 실제 코드/데이터를 수정한다.
- 데이터 파일을 수정했다면 반드시 [data_change_log.md](/home/cwj/llm-project/docs/data_change_log.md)에 기록한다.

즉 AI는 `분석 -> 문서화 -> 검토 -> 반영 -> 기록` 순서를 따라야 한다.

## 작업 순서

### 1. 지표 문제 확인

- 특정 지표에서 실패 패턴을 확인한다.
- 대표 실패 사례와 반복 패턴을 수집한다.

AI 실행 지침:

- 먼저 [eval_results_comparison.md](/home/cwj/llm-project/eval_output/eval_results_comparison.md)를 읽고 어떤 metric이 가장 약한지 확인한다.
- 비교 대상이 여러 모델이면, 절대 성능뿐 아니라 상대적으로 뒤처지는 지표를 찾는다.
- 이 단계의 산출물은 `지금 어떤 metric을 우선 다룰 것인가`에 대한 짧은 요약이다.

### 2. 원인 분해

- 실패를 `eval`, `train data`, `datagen`, `model` 관점으로 나눠 본다.
- GT 문제인지, annotation policy 문제인지, 모델 추론 문제인지 구분한다.

AI 실행 지침:

- [tool_failures_with_dialogue.md](/home/cwj/llm-project/eval_output/qwen-2.5-7b-function-calling_batch2_data_v2/tool_failures_with_dialogue.md) 같은 실패 trace 문서를 읽고, 실패를 유형별로 묶는다.
- 같은 실패라도 모두 구조 문제로 일반화하지 말고, 순수 모델 실수는 따로 분리한다.
- 이 단계의 산출물은 `무엇이 실제 수정 대상이고, 무엇이 단순 모델 실수인가`에 대한 분류다.

### 3. 이슈 단위로 세분화

- 큰 문제를 더 작은 이슈로 나눈다.
- 예: `query/category`, `only_open`, `special_request`, `delivery_note`

AI 실행 지침:

- 하나의 metric failure를 바로 한 문서로 뭉뚱그리지 말고, 반복되는 하위 문제로 쪼갠다.
- 단, 실제 수정 계획이 하나의 규칙으로 수렴하면 문서는 통합해도 된다.
- 이 단계의 산출물은 `이번에 실제로 고칠 하위 문제 목록`이다.

### 4. 보고서 작성

각 이슈마다 `docs/Troubleshooting/...` 아래에 문서를 만든다.

문서에는 최소한 아래 내용을 포함한다.

- 문제가 무엇인지
- 어떤 사례가 근거인지
- 원인이 어디에 있는지
- 해결 방향이 무엇인지
- 실제 개선안이 무엇인지

AI 실행 지침:

- 문서는 단순 요약이 아니라 이후 수정의 기준이 되도록 써야 한다.
- 반드시 아래를 분리해서 적는다.
  - 관측된 실패
  - 실제 수정 가능한 원인
  - 제외할 케이스
  - 추천 규칙
- 사람 검토 전에 실제 코드/데이터를 바꾸지 않는다.

### 5. 문서 커밋

- 문제 정의와 해결안 문서화를 먼저 커밋한다.
- 이 커밋은 “분석과 의사결정”을 남기는 목적이다.

AI 실행 지침:

- 실제 git 커밋은 사용자 요청이 있을 때만 수행한다.
- 다만 작업 단위는 항상 `문서화 단계`와 `실제 반영 단계`를 분리해서 관리한다.
- 사용자가 커밋을 요청하지 않더라도, 답변에서는 이 구분을 유지한다.

### 6. 실제 반영

문서를 근거로 실제 수정 작업을 한다.

- 코드 수정
- 평가 데이터셋 수정
- 훈련 데이터셋 수정
- datagen 규칙 수정

AI 실행 지침:

- 문서에서 합의된 규칙만 반영한다.
- `datagen`, `eval_data`, `train_data` 중 어디를 수정할지 먼저 명시한다.
- 실제 대화 수정이 필요한지, tool schema / system prompt 수정만 필요한지 구분한다.
- `sample` 파일, `dataset`, `train_data` 스냅샷 중 무엇을 건드릴지도 작업 전에 명확히 한다.

### 7. 데이터 변경 기록

데이터 파일이 수정되면 [data_change_log.md](/home/cwj/llm-project/docs/data_change_log.md)에 기록한다.

기록에는 최소한 아래 내용을 남긴다.

- 어떤 문제가 있었는지
- 어떤 기준으로 바꿨는지
- 대표 예시 1개
- 어떤 데이터 파일이 수정되었는지

AI 실행 지침:

- 데이터 수정이 있었다면 이 단계는 생략하지 않는다.
- 로그에는 반드시 `문제`, `변경`, `예시`, `상세 기준`, `변경 파일`을 남긴다.
- 관련 문제 정의 문서를 링크해 추적 가능하게 만든다.

### 8. 반영 커밋

- 실제 코드/데이터 변경은 문서 커밋과 분리해서 커밋한다.
- 이 커밋은 “실제 적용”을 남기는 목적이다.

AI 실행 지침:

- 사용자가 커밋을 요청하지 않았다면, 이 단계는 `커밋 가능한 상태로 정리`하는 데까지 수행한다.
- 사용자가 커밋을 요청하면 `문서화 단계`와 `실제 반영 단계`를 분리해 커밋하는 것을 기본 원칙으로 삼는다.

## AI가 실제로 따를 작업 순서

아래 순서는 AI 에이전트가 실제 작업에서 그대로 따를 기본 순서다.

1. [eval_results_comparison.md](/home/cwj/llm-project/eval_output/eval_results_comparison.md)를 읽고 우선 metric을 고른다.
2. 관련 실패 trace 문서를 읽고 반복 패턴을 모은다.
3. 실패를 `eval / train data / datagen / model`로 분해한다.
4. 순수 모델 실수와 실제 수정 가능한 구조 문제를 구분한다.
5. `docs/Troubleshooting/...` 아래에 문제 정의 문서를 작성하거나 수정한다.
6. 사람 검토를 받아 범위를 조정하고 최종 기준을 확정한다.
7. 확정된 기준에 따라 `datagen`, `eval_data`, `train_data`를 수정한다.
8. 데이터가 바뀌었다면 [data_change_log.md](/home/cwj/llm-project/docs/data_change_log.md)에 기록한다.
9. 필요하면 단계별 데이터 스냅샷이나 git 작업 단위까지 정리한다.

AI는 기본적으로 이 순서를 건너뛰지 말아야 한다. 특히 `문서화 전에 실제 데이터 수정`을 먼저 하는 방식은 예외적인 경우를 제외하고 피한다.

## 권장 커밋 단위

### 1차 커밋

- 문제 정의
- 사례 분석
- 해결 방향
- 개선안 문서화

### 2차 커밋

- 코드 수정
- 평가셋 수정
- 훈련셋 수정
- datagen 수정
- `data_change_log.md` 기록

## 문서 구조 권장안

문제 하나를 다룰 때는 보통 아래 구조를 권장한다.

- 전체 문제 지도 문서 1개
- 개별 이슈 해결 문서 1개씩

예:

- `argument_value_failure_case_report.md`
  - 전체 문제 지도와 우선순위
- `query_category_decision_summary.md`
  - 특정 이슈의 문제 정의와 해결

## 한 줄 요약

이 워크플로우는 아래 두 단계로 요약된다.

- 먼저 문제를 분석하고 문서화한다.
- 그다음 문서를 근거로 코드와 데이터를 수정하고 변경 이력을 남긴다.

AI 관점에서 다시 쓰면:

- 먼저 현재 문제를 구조화하고, 사람과 기준을 맞춘다.
- 그다음 그 기준을 따라 수정하고, 수정 사실을 추적 가능하게 남긴다.
